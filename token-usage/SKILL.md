# Claude Code Token Usage Analyzer

Analyze token usage, costs, and patterns from Claude Code session data stored locally in `~/.claude/projects/`.

## Script Location

The analysis script is co-located with this skill:
```
~/.claude/skills/token-usage/claude_token_usage.py
```

## Usage

Run via (works on any machine/user):

```bash
python "$(echo ~/.claude/skills/token-usage/claude_token_usage.py)" [options]
```

Or on Windows CMD:
```cmd
python "%USERPROFILE%\.claude\skills\token-usage\claude_token_usage.py" [options]
```

### Command Reference

| Command | Description |
|---------|-------------|
| `python claude_token_usage.py` | Console output, last 30 days |
| `python claude_token_usage.py --days 7` | Last 7 days |
| `python claude_token_usage.py --days 90` | Last 90 days |
| `python claude_token_usage.py --all` | All available data |
| `python claude_token_usage.py --date 2026-04-15` | Specific date |
| `python claude_token_usage.py --csv output.csv` | Export to CSV |
| `python claude_token_usage.py --html report.html` | Generate interactive HTML report |
| `python claude_token_usage.py --html report.html --all` | Full HTML report |
| `python claude_token_usage.py --html report.html --days 7` | 7-day HTML report |

### HTML Report Contents

The HTML report includes:
1. **Executive Analysis** — summary paragraph with key metrics
2. **Alerts & Attention Points** — auto-detected anomalies (cost spikes, model concentration, cache issues, budget projections)
3. **Overview Cards** — total tokens, cost, API calls, sessions, peak day/hour
4. **Daily Token Usage** — stacked bar chart (Input/Output/Cache Write/Cache Read)
5. **Daily Cost Trend** — line chart with 7-day moving average
6. **Model Distribution** — doughnut charts + detail table
7. **Hourly Activity** — 24-hour bar chart with peak highlight
8. **Weekday vs Weekend** — comparison cards
9. **Daily Detail Table** — sortable, with totals row

### Opening Reports

After generating HTML, open it in the browser:
```bash
# Windows
start "" "report.html"
# macOS
open report.html
# Linux
xdg-open report.html
```

## Data Source

- **Location**: `~/.claude/projects/**/*.jsonl`
- **Content**: Every Claude Code assistant message includes a `usage` field with `input_tokens`, `output_tokens`, `cache_creation_input_tokens`, `cache_read_input_tokens`
- **Retention**: ~50 days (older session files are auto-cleaned)
- **Scope**: All Claude Code usage on this machine across all projects

## Key Metrics Explained

| Metric | Meaning |
|--------|---------|
| **Input Tokens** | Actual new tokens sent to the model (expensive) |
| **Output Tokens** | Tokens generated by the model (most expensive per-token) |
| **Cache Read** | Tokens served from prompt cache (very cheap, ~1/10 of input price) |
| **Cache Write** | Tokens written to prompt cache |
| **Cache Rate** | (Cache Read + Cache Write) / Total — higher is better, >85% is excellent |
| **I/O Ratio** | Input / Output — high ratio means context-heavy, low-generation workloads |

## Common Questions

**Q: Why is my total token count so high but cost relatively moderate?**
A: Most tokens are Cache Reads, which cost 1/10th of regular input. A 92% cache rate means only ~8% of tokens are billed at full price.

**Q: Why does a day with fewer total tokens cost more?**
A: Check the model mix — a day using 100% Opus costs much more than a day using 40% Sonnet, even with more total tokens. Also check real Input vs Cache Read ratio.

**Q: How to reduce costs?**
A: (1) Route simpler tasks to Sonnet instead of Opus. (2) Use longer sessions to maximize cache hits. (3) Use `/compact` to reduce context size. (4) Avoid frequent context resets.
